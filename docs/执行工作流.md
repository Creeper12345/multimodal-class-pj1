# 执行工作流

这份文档是每次开始运行前都要先看的操作规程。

## 固定原则

1. 先看 [WORKLOG](WORKLOG.md)，确认上一次停在哪。
2. 再看 [PLAN](PLAN.md)，确认这次要推进哪个任务。
3. 运行前先做环境自检，不要直接上全量实验。
4. 先 dry-run，再 smoke test，再全量。
5. 每次跑完后更新 `WORKLOG.md`。
6. 不把数据集、模型权重、缓存提交进仓库。

## 每次运行前的标准顺序

### Step 1. 同步代码

目标：确认服务器上的代码和本地准备的一致。

你要做的事：

- 拉取最新代码
- 确认当前目录结构完整

### Step 2. 激活环境

目标：确认用的是 `multimodal`，不是别的环境。

命令：

```bash
conda activate multimodal
```

### Step 3. 环境自检

目标：确认 Torch、LAVIS、OpenCLIP、数据路径、设备状态正常。

命令：

```bash
python scripts/check_environment.py --check-data
```

通过条件：

- `torch_installed = true`
- CUDA 服务器上 `cuda_available = true`
- 数据路径存在

### Step 4. Dry-run

目标：确认主入口、数据读取、输出路径正常。

命令：

```bash
python code/pj1/task1/run_retrieval.py --dry-run --max-images 10
```

通过条件：

- 能打印图片和 caption 数量
- 不报路径错误

### Step 5. Smoke test

目标：用少量数据测试一个模型能否完整走通。

建议顺序：

1. `lavis:clip_feature_extractor:base`
2. `lavis:blip_retrieval:coco`
3. `lavis:blip2_feature_extractor:pretrain`

### Step 6. 全量运行

目标：在 smoke test 通过后再跑完整任务。

注意：

- 显存不够先减小 `--image-batch-size` 和 `--text-batch-size`
- 若模型已经下载过，不要轻易 `--overwrite-cache`

### Step 7. 记录

目标：让下一轮工作有上下文。

需要记录到 `WORKLOG.md` 的内容：

- 运行日期
- 运行命令
- 成功还是失败
- 结果文件路径
- 失败时的关键报错
- 本轮做了什么修复

## 遇到报错时的处理方式

### 环境报错

优先回传：

- `python scripts/check_environment.py --check-data`
- 安装命令
- 完整报错

### 模型下载报错

优先回传：

- 哪个模型规格失败
- 下载到哪一步
- 缓存目录里有哪些 partial 文件

### 显存不足

优先处理：

- 把 `--image-batch-size` 降到 `1` 或 `2`
- 把 `--text-batch-size` 降到 `8` 或 `16`
- 保留 `--eval-batch-size 128` 或更小

## 当前推荐的运行节奏

1. 环境自检
2. dry-run
3. CLIP smoke test
4. BLIP retrieval smoke test
5. BLIP-2 smoke test
6. 三模型全量 Task 1
7. 记录结果并准备 Task 2
